Implement a simple Packing / Dispatch module in FastWMS as the next outbound operational step after picking.

GOAL
Add a practical packing and dispatch workflow so picked waves/orders can be:
- consolidated
- packed
- assigned to parcels/packages
- marked as ready for dispatch
- closed with traceability

IMPORTANT
This is a simple operational packing/dispatch module for V1.
Do NOT build a full transport management system.
Do NOT implement complex carrier integrations yet.

This module must work for:
- Planner/store-distribution waves
- MANUAL_WEB waves
- MANUAL_OTHER waves

==================================================
PART 1 — BUSINESS SCOPE
==================================================

We need an outbound workflow after picking:

1) Picking finishes
2) Order/folio/wave enters packing stage
3) User can consolidate picked units into packages
4) Packages can be closed
5) Run can be marked ready for dispatch / dispatched

This must support two main realities:
- Store distribution folios
- WEB/manual orders

==================================================
PART 3 — PACKING RUN CREATION
==================================================

Create module/pages:
- /wms/packing
- /wms/packing/new
- /wms/packing/<packing_run_id>

Permissions:
- warehouse_manager or admin
- or permission such as wms:pack_manage

Packing run can be created from:
1) completed pick wave
2) completed planner folio
3) manual WEB wave
4) manual OTHER wave

Rules:
- if wave is not PICKED / CLOSED, packing cannot start
- use actual picked quantities as source
- never pack more than actually picked

==================================================
PART 4 — PACKING RUN BUILD LOGIC
==================================================

Create service:
create_packing_run_from_wave_or_plan(...)

Behavior:
1) Read actual picked quantities from WMS tasks/reservations
2) Aggregate by SKU
3) Create WmsPackingRun
4) Create WmsPackingLine rows:
   qty_picked = actual picked
   qty_packed = 0
   qty_pending = qty_picked
   status = OPEN

Packing type mapping:
- Planner/store folio -> STORE_TRANSFER
- MANUAL_WEB -> WEB_ORDER
- MANUAL_OTHER -> MANUAL

==================================================
PART 6 — PACKAGE ASSIGNMENT LOGIC
==================================================

1) User can create one or more packages per packing run
2) For each package:
- add SKU quantities from open packing lines
- decrement qty_pending
- increment qty_packed
- update packing line status:
   OPEN / PARTIAL / PACKED

3) Rules:
- cannot pack more than qty_pending
- cannot close run while qty_pending > 0 unless business chooses partial shipping
For V1:
- allow READY_TO_DISPATCH only when all qty_pending = 0

==================================================
PART 7 — DISPATCH STATUS FLOW
==================================================

Packing run status flow:
- DRAFT
- IN_PROGRESS
- PACKED
- READY_TO_DISPATCH
- DISPATCHED
- CLOSED

Rules:
1) once first package activity occurs -> IN_PROGRESS
2) when all qty_pending = 0 -> PACKED
3) supervisor/user can mark -> READY_TO_DISPATCH
4) then -> DISPATCHED
5) optional final -> CLOSED

==================================================
PART 8 — INTEGRATION WITH EXISTING FLOWS
==================================================

1) Planner/store distribution folios
- after picking is complete and folio is in Empacado, packing run can be created
- packing run can serve as the operational bridge before final dispatch/close if useful

2) WEB/manual waves
- after wave is picked, packing run can be created
- this is especially important for ecommerce orders

3) Existing folio close/store transfer logic
Do NOT break it.
If needed, packing can initially be informational/operational without replacing final stock transfer closure yet.
But prepare structure so dispatch can later integrate cleanly.

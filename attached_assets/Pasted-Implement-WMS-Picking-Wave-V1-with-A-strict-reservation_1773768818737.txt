Implement WMS Picking Wave V1 with:
A) strict reservation priority logic by location
B) operator issue reporting during picking execution

CONTEXT
FastDSO + FastWMS already exist in a single Flask app.
We need picking waves to be created from approved plans/folios and executed by operators.
This version uses:
- single warehouse: MAIN
- picking consolidated by SKU per folio
- no store separation in picking stage
- exact reservation breakdown by location
- operator must be able to report why a SKU could not be fully picked

PART A — PICK WAVE RESERVATION PRIORITY

BUSINESS RULE
For each SKU in a pick wave, reserve inventory from eligible locations in this exact order:

1) location_type priority:
   - PICK first
   - BULK second
2) inside the same type:
   - lower available_units first
3) if tie:
   - location_code ascending

ELIGIBILITY
A location is eligible only if:
- warehouse_code == "MAIN"
- is_active == true
- available_units > 0

Where:
available_units = on_hand_units - allocated_units

OBJECTIVE
This policy should empty small locations first and reduce stock fragmentation.

RESERVATION SERVICE
Create or update service function:

create_pick_wave_from_plan(plan_id, warehouse_code="MAIN", created_by_user_id=None)

Behavior:
1) Aggregate requested quantities by SKU from approved plan lines
2) For each SKU:
   - required_qty = sum(plan line qty_final, or qty_suggested if qty_final not used)
   - fetch all eligible inventory rows for that product in MAIN
   - compute available_units
   - sort rows by:
       a) location_type_priority (PICK=1, BULK=2)
       b) available_units ascending
       c) location_code ascending
   - reserve until required_qty is met:
       reserve_qty = min(remaining_qty, available_units)
       increment inventory.allocated_units by reserve_qty
       create WmsPickReservation
       remaining_qty -= reserve_qty
3) Create one WmsPickTask per SKU:
   - requested_qty = required_qty
   - picked_qty = 0
   - short_qty = remaining_qty if any
   - status:
       OPEN if fully reserved
       SHORT if partially reserved or zero reserved
4) Wave status:
   - READY_TO_ASSIGN if all SKUs fully reserved
   - NEEDS_REVIEW if any SKU has short_qty > 0
5) Use one DB transaction. On error rollback everything.
6) Never allow allocated_units > on_hand_units.

BUSINESS GOAL

PICK EXECUTION LOGIC
We are not building the full mobile UI yet, but backend must support execution and issue reporting now.

Create/update these endpoints:

1) POST /api/wms/waves/<wave_id>/start
- only assigned operator or manager can start
- if wave status == ASSIGNED, set:
   status = IN_PROGRESS
   pick_started_at = now()

2) POST /api/wms/tasks/<task_id>/pick
Input JSON:
{
  "picked_qty": int,
  "note": optional
}
Behavior:
- validate picked_qty >= 0
- picked_qty cannot exceed requested remaining qty for task
- update WmsPickTask.picked_qty += picked_qty
- update task.status:
   DONE if picked_qty total == requested_qty and no shortages
   IN_PROGRESS otherwise
- allocate picked_qty proportionally across OPEN reservations in order of location priority already stored
- for each affected reservation:
   increment reservation.picked_qty
   if reservation.picked_qty == reservation.reserved_qty -> status PICKED
   else -> status PARTIAL
- decrement WmsInventory.on_hand_units by picked amount
- decrement WmsInventory.allocated_units by picked amount
- update wave picked_units / picked_lines counters

3) POST /api/wms/tasks/<task_id>/report-issue
Input JSON:
{
  "missing_qty": int,
  "reason_code": "...",
  "operator_note": "..."
}
Behavior:
- missing_qty must be > 0
- create WmsPickIssue rows for the relevant reservation(s) / location(s)
- increment task.short_qty by missing_qty
- update task.status:
   SHORT if picked_qty + short_qty >= requested_qty
   else IN_PROGRESS
- update reservation statuses to SHORT or PARTIAL as needed
- release unpicked allocated stock for the affected reservation:
   reservation short means the remaining reserved-but-not-pickable qty should be removed from allocated_units
   IMPORTANT: do NOT decrease on_hand_units for missing qty, only decrease allocated_units because it was not physically picked
- if reason_code in (DAMAGED, QUALITY_HOLD, MIXED_SKU):
   optionally create a WMS inventory event/log flag for investigation
- update wave short_units / short_lines counters

4) POST /api/wms/waves/<wave_id>/finish
Behavior:
- wave can be finished only if all tasks are DONE or SHORT
- set:
   pick_finished_at = now()
   status = PICKED
- manager/admin can later close it:
   status = CLOSED
   closed_at = now()

STATE RULES
Task status logic:
- OPEN: nothing picked yet, no issue reported
- IN_PROGRESS: partial picked, still pending
- DONE: picked_qty == requested_qty
- SHORT: requested_qty cannot be fully picked and issue(s) recorded

Reservation status logic:
- OPEN: reserved but untouched
- PARTIAL: partially picked
- PICKED: fully picked
- SHORT: issue reported against remaining reserved qty
